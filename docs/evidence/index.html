2. API Validation: Critique Mode Accepted/Rejected

The backend validates critique_mode on PUT /api/settings and returns 400 for invalid values.

PUT /api/settings {"critique_mode": "freeform"} → 200 OK critique_mode=freeform

4. Architecture: Phase 2 Data Flow

How claim-level critique mode works across the 3-stage pipeline:

Round 1: Stage 1: Models respond independently (same as freeform) | Claim Extraction: Chairman decomposes responses into canonical claims | e.g. A1: "Quantum computers use qubits in superposition" | A2: "Current quantum computers have 10M qubits" | B1: "Error correction is the primary bottleneck" | Stage 2: Peers evaluate canonical claims (not free-text) | A1: STRONG (3/4 agree) | A2: FLAWED (4/4 agree) | B1: STRONG | Stage 3: Chairman synthesizes (unchanged) Round 2: Per-model prompts: Model A sees: "Your A2 was FLAWED. Consider B1 from Model B (STRONG)." Model B sees: "Your B1 was STRONG. Consider A1 from Model A (STRONG)." | Stage 1: Models revise with targeted feedback | [Repeat Stage 2 + 3, check convergence]
